- [MED] **`review-gate.sh` manual post case lacks `*)`/else branch** (`docs/harness/bin/review-gate.sh:467–481`): After `write_manual_review` + `parse_verdict`, the `case "$VERDICT"` block handles `PASS` and `FAIL` but has no `*)` arm. In practice this is dead code — `write_manual_review` guarantees REVIEW_VERDICT exists before saving, and `write_review` injects a synthetic `REVIEW_VERDICT: FAIL` when none is found — so `parse_verdict` always returns `PASS` or `FAIL`. But if execution somehow falls through (e.g., future refactor of `write_review`), the script silently drops to the `codex` branch at line 483, running a live automated review under `REVIEWER_CLI=manual`. A defensive `*) exit 1` arm or an explicit `exit 1` after the case block would close this latent path.

- [MED] **`check-commit-msg.sh` silently ignores unrecognized bare arguments** (`docs/harness/bin/check-commit-msg.sh:20–26`): The `*)` branch in the arg-parsing loop accepts non-file paths silently (they're `shift`'d without error). E.g., `check-commit-msg.sh --message "feat(adapter): x" --typo-flag` returns exit 0 without warning. The canvas documents this as by-design for hook invocation, but a CLI user could believe a flag was processed when it was silently dropped. Consider at minimum emitting a warning to stderr for non-file, non-`--message` arguments. No hook-safety regression since git passes only the COMMIT_EDITMSG path.

- [LOW] **`doctor.sh` JSON escape is incomplete for control characters** (`docs/harness/bin/doctor.sh:275`): The `fail()` accumulator escapes `\` and `"` only. If a future failure message contains a tab or newline, the emitted JSON would be invalid. Current `require_*()` labels are short identifiers with no control characters, so there is no present defect. `sensors.sh` already implements a `json_escape()` helper covering tabs, CR, and newlines — importing or calling that helper in `doctor.sh` would close the gap proactively.

- [LOW] **`prior_unresolved_findings` includes the `*-pre-codex.md` artifact when it is the most recent file for a task** (`docs/harness/bin/review-gate.sh:270`): The find pattern `*-$(task_slug)-post-*.md` correctly scopes to post artifacts, so pre artifacts are not re-injected. No defect, but worth noting the filter is only on the basename suffix (`-post-`), not on the full naming convention; a future reviewer tooling change that writes a file matching the post pattern for pre runs would be re-injected unintentionally.
